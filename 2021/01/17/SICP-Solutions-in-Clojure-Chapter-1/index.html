Space complexity

It’s easy to observe the space complexity is O(max{m,n})O(max\{m, n\})O(max{m,n}).

Time complexity

Let’s suppose n be the money to be changed, m be the count of type of coins, VmV_mVm​ be the denomination of the coin of mth type.

The following is only a rough estimate.

T(n,m)=∑i=0⌈nVm⌉T(n−iVm,m−1)+O(1)=O(⌈nVm⌉T(n,m−1))=O(∏i=1m⌈nVm⌉T(n,0))=O(nm)\begin{aligned} &T(n, m) &=& \sum_{i=0}^{\lceil\frac{n}{V_m}\rceil}T(n-iV_m, m-1)+O(1)& \\ &&=&O(\lceil\frac{n}{V_m}\rceil T(n, m-1))& \\ &&=&O(\prod_{i=1}^{m}\lceil\frac{n}{V_m}\rceil T(n, 0))& \\ &&=&O(n^m)& \end{aligned} ​T(n,m)​====​i=0∑⌈Vm​n​⌉​T(n−iVm​,m−1)+O(1)O(⌈Vm​n​⌉T(n,m−1))O(i=1∏m​⌈Vm​n​⌉T(n,0))O(nm)​​

For m = 5, there there is a closer estimate, implying that this algorithm has a time complexity of Θ(n5)\Theta(n^5)Θ(n5) in codology.net.

Optimization on time complexity

If you examine the “tree” given before, you’ll realize that it’s actually not a tree. It’s a topological graph.

In the original tree, there is a lot of repeating tree nodes, calculating the same value of the recursive function.

If we save the value of every calculated function, the time complexity will be O(nm)O(nm)O(nm) , since there are only n×mn \times mn×m states in total.

For Clojure, using memoize will simply complete the job for you.

However, the space complexity will fall to O(n×m)O(n \times m)O(n×m) , but usally the optimization as a whole is a great deal.

This kind of way of solving problems are actually Dynamic programming. Notice: This kind of optimization only works for referentially transparent functions, that is, the function’s output only relates to its input.
